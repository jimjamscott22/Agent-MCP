Agent-MCP Implementation Plan

Human-in-the-Loop Memory Updates & Web UI Integration

This document outlines the technical strategy for extending Agent-MCP with a human-in-the-loop memory curation system. By shifting away from autonomous, direct file writes, this architecture allows agents to propose context updates (rules, architectural decisions, solved bugs) which are then queued for manual user approval via the local Admin Web UI.

1. The Proposal Pipeline Architecture

Currently, the AgentsRegistry supports direct file editing. We will introduce a staging layer called the Proposal Queue. This queue will temporarily hold agent-generated markdown chunks until the user formally approves them, at which point they are merged into the target AGENTS.md file.

Architectural Shift

Instead of update_managed_file_section, agents will exclusively use a new tool: propose_registry_update. The server will write these proposals to a local SQLite database or JSON file (e.g., .cache/memory_proposals.json) and instantly surface them in the Web UI.

2. Data Models & Schemas

A new internal model is required to track the lifecycle of a memory proposal.

from pydantic import BaseModel
from typing import Optional, Literal
from datetime import datetime

class MemoryProposal(BaseModel):
    id: str                    # Unique UUID
    target_project: str        # e.g., "frontend" or root path
    section_heading: str       # e.g., "Coding Rules", "Notes"
    proposed_content: str      # The markdown payload
    rationale: str             # Why the agent thinks this should be saved
    status: Literal["pending", "approved", "rejected"] = "pending"
    created_at: datetime
    agent_id: Optional[str]    # Ties into the Inventory system

3. MCP Tool Modifications

MCP Tool propose_registry_update

Purpose: Allows the agent to submit a draft update without modifying the filesystem.

  • Inputs: target_project, section_heading, proposed_content, rationale.
  • Logic: Validates the target project exists in the registry. Appends the MemoryProposal to the local queue.
  • Return to Agent: "Proposal [ID] successfully submitted. Awaiting human approval."

System Prompt Injection

Purpose: Conditions the agent to use the new workflow.

We will dynamically prepend this global instruction to the root AGENTS.md resolution:

SYSTEM DIRECTIVE: 
When you resolve a complex bug or establish a new rule, DO NOT assume it should be saved.
Ask the user: "Should I add this to the project's AGENTS.md?"
If the user agrees, use the `propose_registry_update` tool. Do not use direct file write tools.

4. Web UI Extensions (FastAPI + JS)

The local admin interface (currently at port 8002) is the core of the human-in-the-loop experience. We will add a "Memory Proposals" dashboard tab.

New API Endpoints

  • GET /api/proposals - Fetch all pending proposals.
  • POST /api/proposals/{id}/approve - Triggers the actual file write logic (reuses existing update_managed_file_section backend logic) and marks status as approved.
  • POST /api/proposals/{id}/reject - Marks as rejected and archives.
  • PATCH /api/proposals/{id} - Allows the human to edit the proposed markdown before approving.

Frontend UX (app.js / admin.html)

The UI will feature a notification badge next to a "Review Queue" tab. Each proposal will render as a card displaying:

  1. Target: Badge indicating which AGENTS.md file will be modified.
  2. Rationale: The agent's reasoning.
  3. Diff Viewer: A markdown text area where the user can tweak the code before hitting "Merge".

5. Execution Action Items

Follow these steps to implement the workflow in the Python codebase:

  • Data Layer: Create mcp_agents_registry/proposals.py with the Pydantic schema and JSON file-backed CRUD operations.
  • Server Layer (MCP): Register @mcp.tool() def propose_registry_update(...) in server.py.
  • Prompt Layer: Update resolver.py to prepend the System Directive string to the top of resolved context payloads.
  • Web Layer (Backend): Add the four new /api/proposals routes in web.py.
  • Web Layer (Frontend): Update admin.html and app.js to include the Proposal Review panel.
  • File Writing: Link the approve endpoint to the existing update_managed_file_section() utility, ensuring the file's mtime updates correctly trigger an automatic cache invalidation.

Bonus: Proactive Context Refresh (Watchdog)

To eliminate polling delays after a proposal is approved, integrate the watchdog Python library in scanner.py. This will instantly detect the human-approved write to AGENTS.md and re-parse the hierarchical context graph in real-time.
