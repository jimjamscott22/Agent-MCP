This document outlines the technical strategy for extending Agent-MCP with a human-in-the-loop memory curation system. By shifting away from autonomous, direct file writes, this architecture allows agents to propose context updates (rules, architectural decisions, solved bugs) which are queued for manual user approval via the local Admin Web UI.

1. The Proposal Pipeline Architecture

The AgentsRegistry supports managed file editing. The proposal feature adds a staging layer called the Proposal Queue. This queue holds agent-generated markdown chunks until the user approves them, at which point they are merged into the target AGENTS.md or compatible managed instruction file.

Architectural Shift

Agents should use propose_registry_update for memory updates. Direct write tools are available only when allow_direct_writes: true. The proposal queue is JSON-backed at .cache/memory_proposals.json, matching the current repo style and avoiding a new SQLite dependency.

2. Data Models & Schemas

The proposal model uses @dataclass, matching the existing model style in this repository. Pydantic is not required for this feature.

@dataclass(slots=True)
class MemoryProposal:
    id: str                    # UUID4 string
    target_project: str        # project_name from registry
    target_path: str           # absolute AGENTS.md/CLAUDE.md path to modify
    section_heading: str       # e.g. "Coding Rules", "Notes"
    proposed_content: str      # markdown body for the section
    rationale: str             # why the agent thinks this should be saved
    status: str                # "pending", "approved", or "rejected"
    created_at: str            # ISO-8601 UTC
    resolved_at: str | None    # ISO-8601 UTC or None
    agent_id: str              # optional inventory/agent identifier

Storage Requirements

  • Persist proposals to .cache/memory_proposals.json.
  • Use a JSON envelope: {"version": 1, "proposals": [...]}.
  • Use atomic writes and add read-modify-write locking so concurrent proposal submissions cannot drop data.
  • Only allow mutable fields to be changed after creation: section_heading, proposed_content, and internal status fields.

3. MCP Tool Modifications

MCP Tool propose_registry_update

Purpose: Allows the agent to submit a draft update without modifying the filesystem.

  • Inputs: target_project, section_heading, proposed_content, rationale, optional agent_id.
  • Logic: Validate the target project exists, resolve ProjectRecord.agent_file_path, validate that path through the managed-file path guard, then append a MemoryProposal to the local queue.
  • Return to Agent: {"proposal_id": "...", "status": "pending", "message": "Proposal ... submitted. Awaiting human approval."}

System Directive

Purpose: Conditions the agent to use the new workflow.

Expose this directive as both an MCP prompt and a resource, and inject it into resolved context payloads so agents receive the instruction during normal resolve_context flows.

SYSTEM DIRECTIVE: 
When you resolve a complex bug or establish a new rule, DO NOT assume it should be saved.
Ask the user: "Should I add this to the project's AGENTS.md?"
If the user agrees, use the `propose_registry_update` tool. Do not use direct file write tools.

Direct Write Gate

  • allow_direct_writes: false is the default.
  • When false, update_managed_file and update_managed_file_section are not registered as MCP tools.
  • Approval still calls the underlying registry write method directly after a human approves the proposal.

4. Web UI Extensions (FastAPI + JS)

The local admin interface is the core of the human-in-the-loop experience. Add and maintain a Review Queue dashboard tab for pending and historical memory proposals.

New API Endpoints

  • GET /api/proposals - List proposals. Supports optional ?status=pending|approved|rejected.
  • POST /api/proposals - Convenience/admin endpoint for creating a proposal through the same validation path as the MCP tool.
  • POST /api/proposals/{id}/approve - Triggers the actual file write logic, refreshes the index, and marks status as approved.
  • POST /api/proposals/{id}/reject - Marks as rejected and archives.
  • PATCH /api/proposals/{id} - Allows the human to edit section_heading and proposed_content before approving. Must reject edits unless status is pending.

API errors should use a consistent JSON shape. The working target is {"error": "..."} for proposal routes unless the whole admin API is deliberately standardized around another shape.

Frontend UX (app.js / admin.html)

The UI will feature a notification badge next to a "Review Queue" tab. Each proposal will render as a card displaying:

  1. Target: Project badge plus target file path, truncated visually with a tooltip containing the full path.
  2. Agent ID: Shown when non-empty.
  3. Rationale: The agent's reasoning.
  4. Review editor: Editable heading input and markdown textarea for pending proposals.
  5. Actions: Approve and Reject buttons. After success, remove the card from the pending queue and refresh the badge.
  6. History: Toggle to show approved/rejected proposals as read-only cards, including their final content.

Phase 2: Proactive Context Refresh (Watchdog)

To eliminate polling delays for external edits and non-proposal writes, integrate the watchdog Python library around configured roots. Approval-triggered writes already control the write path and should refresh immediately, but watchdog support can keep the registry current when users or other tools edit AGENTS.md/CLAUDE.md outside the proposal flow.
